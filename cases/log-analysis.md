# Case: log-analysis

## Domain

Server and application logs are the canonical "semi-structured text" workload: each line follows a loose convention (timestamp, severity, service, message) but the message payload is free-form and vendor-specific. Extracting structured summaries — error counts by severity, top recurring error patterns, services with the most failures — is an everyday ops task. It appears in on-call triage, post-incident reviews, SLO dashboards, and "what broke overnight" morning routines.

The task sits squarely in Bricks' sweet spot on paper: the work is mechanical (regex extract → group → count → sort → top-K), the volume is unbounded, and the output shape is small and deterministic. The question is whether Bricks' brick catalog covers the composition cleanly, and where the cost curve crosses the raw-LLM baseline.

## Task specification

**Input:** a list of log lines (strings), each roughly shaped `"<ISO-timestamp> <SEVERITY> <service> - <message>"` but with intentional noise (extra whitespace, occasional missing fields, rare malformed lines).

**Output:** a JSON object with two keys:

```json
{
  "severity_counts": {
    "ERROR": 42,
    "WARN": 108,
    "INFO": 850,
    "DEBUG": 0
  },
  "top_error_patterns": [
    {"pattern": "Connection refused to upstream", "count": 18},
    {"pattern": "Timeout after 30s calling /api/users", "count": 12},
    {"pattern": "Null pointer in OrderService.process", "count": 7}
  ]
}
```

**Definitions:**
- `severity_counts` — exact count per severity level across all parseable lines. Unparseable lines are ignored (not an error).
- `top_error_patterns` — the top 3 distinct `message` strings among lines with severity `ERROR`, ordered by count desc, ties broken by lexical order of the pattern.

**Success criteria:**
- `severity_counts` must match ground truth exactly (integer equality per key).
- `top_error_patterns` must match ground truth as an ordered list (pattern + count), length 3.

Partial credit: per-field diffs recorded in `verdict.per_field_diffs` but the run's `*_correct` flag is strict all-or-nothing.

## Data

Synthetic, generated by `cases/log-analysis/generate.py`. Seeded RNG, five sizes: 50, 200, 1000, 5000, 20000 lines. Generator also emits ground truth as a sibling `.truth.json` file so verification is a file compare, not a re-derivation.

Distribution (per size):
- 80% INFO, 12% WARN, 7% ERROR, 1% DEBUG (roughly; sampled, not exact)
- ~5% of lines are "noisy": extra whitespace, lowercase severity, or truncated
- ~1% are malformed beyond recovery (no severity token at all) — should be skipped silently
- ERROR messages are drawn from a fixed pool of 12 templates with Zipf-ish frequency so the top-3 is stable and non-trivial to guess

Seed data file: `cases/log-analysis/data-<size>.json` (list of strings).
Ground truth: `cases/log-analysis/data-<size>.truth.json`.

## Hypothesis

**Bricks wins on cost past ~500 lines.** Raw LLM token cost scales linearly with input size (each log line is ~30–60 tokens); Bricks pays LLM cost once for compose, then runs deterministically. Crossover expected between 200 and 1000 lines on sonnet.

**Bricks wins on reliability at all sizes ≥ 200.** Raw LLM will likely miscount at scale — counting is a known LLM weakness, and 1000+ items guarantees at least one off-by-N error on `severity_counts`. Bricks, if it composes correctly once, is exact forever after.

**The compose step is the risk.** `top_error_patterns` requires: filter-by-severity → group-by-message → count → sort-desc → take-3 → format. This is 5+ bricks chained. If the stdlib is thin on group-by or sort-by-count, Bricks may need fallback Python, which defeats the cost argument. I expect to surface at least one missing or awkward brick here — a candidate for `proposals/`.

**Noisy/malformed lines are a wild card.** Raw LLM handles them gracefully (it just "knows" what a log line looks like). Bricks needs explicit parsing logic; if the compose step over-trusts structure, it'll crash on malformed input or silently miscount. Worth watching.

**Secondary expectation:** haiku on raw-LLM will be cheaper than sonnet but less reliable at large N; opus will be expensive and may not out-perform sonnet meaningfully. Start with sonnet only; add haiku if the cost-curve story benefits from a cheaper baseline.
