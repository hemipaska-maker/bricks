# Case: numeric-stats

## Domain

Aggregate statistics over a numeric list — sum, mean, min, max, count. The simplest non-trivial structured-data task: pure reduction, no filtering, no grouping, no sorting. Chosen deliberately as a Bricks "best case" — if compose can't survive this, something is broken catalog-wide; if it can, we have a working pair for the cost-curve experiment.

## Task specification

**Input:** a list of floats under `inputs.values`.

**Output:**

```json
{"sum": 123.45, "mean": 12.345, "min": 0.5, "max": 98.7, "count": 10}
```

All values as numbers. Count is integer, rest are floats. Ties broken by standard arithmetic.

**Success:** all five fields match ground truth within `abs(expected - got) < 1e-6` for floats, exact for count.

## Data

Generated by [cases/numeric-stats/generate.py](numeric-stats/generate.py). Sizes 50, 200, 1000, 5000, 20000. Values drawn from a mixed uniform + lognormal so tails aren't degenerate. Seed 42.

## Hypothesis

**Bricks wins on cost almost immediately.** Raw LLM input tokens scale linearly with N (each float is ~5-10 tokens as text); at N=5000 we're pushing ~25k-50k input tokens per call. Bricks pays one compose (expensive, but simple task → low output tokens) then runs deterministically with 0 marginal LLM tokens.

**Bricks is exact; raw LLM is not.** LLMs are known to miscompute sums of long lists (attention dilutes; they approximate). Expect raw sonnet to be off by small amounts at N≥200, larger at N≥1000. Bricks, once composed, is arithmetic — exact forever.

**Compose should succeed.** The task maps to a straight sequence (sum → divide → min → max → len). No `for_each` with complex lambdas, no branching. This is the weakest stress test for the composer we can design.

**Tipping point guess:** N ≈ 100 on cost; reliability crossover potentially at N = 1.
